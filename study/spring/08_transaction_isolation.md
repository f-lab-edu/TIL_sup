# 트랜잭션 격리 수준 - isolation
여러 트랜잭션이 동시에 실행될 때, 서로의 변경 내용을 어디까지 볼수 있게 할지 정하는 규칙이다.  
트랜잭션 격리 수준은 동시에 실행되는 여러 트랜잭션 간의 상호작용을 제어하여 동시성 문제를 방지하는데 중요한 역할을 한다. 격리 수준이 높을수록 트랜잭션 간의 간섭을 최소화하여 데이터 정합성을 보장할 수 있지만, 반대로 성능 저하를 초래할 수 있다. 따라서 시스템의 요구사항에 따라 적절한 격리 수준을 선택해야 한다.

!!! info "왜 격리 수준이라 할까?"
트랜잭션을 다른 트랜잭션으로부터 얼마나 격리할지의 정도를 나타내기 때문에 수준(level) 이라고 부른다.

트랜잭션의 ACID에서 I가 Isolation(격리성)이다.
여기서 격리성은 **동시에 실행되는 다른 트랜잭션의 영향을 얼마나 차단할 것인가**를 나타낸다.

그런데 이 격리는 있다/없다가 아닌 강도를 여러 단계로 나눌 수 있다.

- Read Uncommited (레벨 0)
- Read Commited (레벨 1)
- Repeatable Read (레벨 2)
- Serializable (레벨 3)

이걸 격리의 단계, 즉 격리 수준이라고 부른다.

## 격리 수준을 알아야 하는 이유
격리 수준은 동시성과 데이터 정합성 사이의 균형을 조절하기 위한 기준이라고 볼 수 있다.

트랜잭션이 동시에 실행된다면 다음과 같은 문제가 발생한다.

계좌 잔액이 10000원 있다.
- 트랜잭션 A가 1000원을 추가함
- 아직 커밋 안함
- 트랜잭션 B가 11000원을 읽음
- 트랜잭션 A가 롤백

그러면 B는 롤백되지 않은 값을 읽어 문제가 발생한다. 이를 Dirty Read라고 하는데 이외에 다른 문제도 있다.

## 동시성 문제

### Dirty Read
커밋되지 않은 데이터를 다른 트랜잭션이 읽는 문제

1. 트랜잭션 A가 1자리 남은 예약을 시도한다. 남은 예약좌석은 0이 된다.
2. 트랜잭션 B가 예약 가능 좌석이 0인걸 확인하고 종료한다.
3. 트랜잭션 A가 카드 한도 초과로 실패한다. 남은 예약좌석이 1이 된다.

결과적으로 예약 좌석이 1자리가 남았음에도 불구하고, 트랜잭션 B는 예약을 할 수 없게 된다.

### Non-repeatable Read
같은 트랜잭션 안에서 같은 데이터를 두 번 읽었는데 값이 달라지는 문제

- 트랜잭션 A가 상품 재고 10 조회
- 트랜잭션 B가 재고를 8로 수정 후 커밋
- 트랜잭션 A가 다시 조회하여 8

이렇듯 같은 트랜잭션 안에서도 조회 결과가 달라진다.

### Phantom Read
같은 조건으로 조회했는데, 처음엔 없던 행이 나중에 생기거나 사라지는 문제

- 트랜잭션 A가 price > 1000인 상품 목록 조회
- 트랜잭션 B가 조건에 맞는 새 상품 추가 후 커밋
- 트랜잭션 A가 다시 같은 조건으로 조회
- 처음엔 없던 행이 나타남

즉, 행 자체가 유령처럼 나타난다 해서 phantom이라 한다.

## 격리 수준 종류

### Read Uncommited

- 커밋되지 않은 데이터를 읽을 수 있다.
- 데이터 정합성이 보장되지 않으므로 Dirty Read 문제가 발생할 수 있다.
- Non-reapeatable Read 허용
- Phantom Read 허용

장점
- 동시성이 가장 높다.
- 락 부담이 적다.

단점
- 정합성이 매우 약하다.
- 실무에서는 거의 사용하지 않는다.

### Read Committed

- 커밋된 데이터만 읽을 수 있어 Dirty Read를 방지할 수 있다.
- Non-Repeatable Read 허용
- Phantom Read 허용

장점
- Diry Read 방지
- 성능과 정합성의 균형이 좋다.

단점
- 같은 트랜잭션 안에서 다시 조회하면 값이 바뀔 수 있다.

### Repeatable Read

동일 데이터를 반복 조회할 때 항상 같은 값을 읽어 일관성을 강하게 보장한다.
- Dirty Read 방지
- Non-repeatable Read 방지
- 데이터의 추가나 삭제로 인해 Phantom Read가 발생할 수 있다.

장점
- 같은 행을 반족 조회할 때 값이 바뀌지 않는다.
- 정합성을 더 높게 보장한다.

단점
- 락이나 MVCC 부담이 커질 수 있다.
- DB 구현체에 따라 동작 차이가 있다.

### Serializable

- 트랜잭션이 순차적으로 실행되는 것처럼 동작
- Dirty Read, Non-Repeatable Read, Phantom Read를 모두 방지한다.

장점
- 정합성이 가장 강함

단점
- 동시성이 가장 낮다.
- 락 경합, 대기, 성능 저하 가능성이 크다.

### 정리
격리 수준이 높아질수록
- 정합성은 좋아짐
- 동시성은 떨어짐
- 락 대기나 성능 저하 가능성이 증가

즉, 문제 상황에 따라 맞게 선택해서 사용해야 한다.

- 일반 조회/수정: Read Commited
- 같은 행 반복 조회 : Repeatable Read
- 선착순 쿠폰 : Serializable 검토

다만 실제 동시성 문제는 격리 수준만으로 해결되지 않은 경우가 많아서 다음을 함께 고려해야 한다.
- 비관적 락
- 낙관적 락
- unique 제약조건
- 원자적 update 쿼리

즉, 격리 수준은 동시성 제어의 한 축이다.

## DBMS 별 default 격리 수준
또한, 격리 수준은 DBMS 별로 default 값이 다르다. 그래서 DBMS 별로 격리 수준이 어떻게 설정 되어있는 지 알아두어야 한다.

### MySQL
MySQL의 경우 버전에 관계없이 InnoDB 스토리지 엔진을 사용하는 한  REPEATABLE READ가 기본 격리 수준으로 유지하고 있다.

이는 다른 DBMS와 달리 두드러지는 차이점인데, MySQL이 REPEATABLE READ를 고수하는 이유는 복제(Replication)의 안정성 때문이다.

- **과거 복제 방식** : 예전 MySQL은 실행된 SQL 문 자체를 복제 서버로 보내는 방식(Statement Based REplication)을 사용했다. 이때 격리 수준이 낮으면 마스터와 슬레이브 간의 데이터 순서가 꼬여 데이터가 달라질 위험이 있다.
- **Next-Key Lock** : MySQL에서 지원하는 기능 중 하나로 REPEATABLE READ에서 발생할 Phantom Read 현상을 거의 완벽하게 방어해준다. 즉, 성능 대비 데이터 정합성이 좋다.

!!! info "실무 팁"
현재는 다른 DB와 호환성을 맞추기 위해 READ Committed로 격리 수준을 낮추고, Replication(RBR)방식을 사용하는 추세라고 한다. MySQL 5.7.7 이상부터는 복제 작업시 SBR이 아닌 RBR 방식을 사용한다.


### Oracle
Oracle의 기본 값은 READ COMMITED이다. 오라클은 MVCC 기술로 높은 동시성을 보장한다.

#### 특징
- 커밋된 데이터만 노출 : 다른 트랜잭션이 수정을 완료하고 COMMIT을 하지 않으면 수정된 데이터를 읽지 못한다.
- 문장 단위의 일관성(Statement-Level Read Consistency) : 하나의 쿼리가 시작된 시점을 기준으로 데이터를 읽는다. 쿼리가 실행되는 도중에 다른 트랜잭션이 데이터를 커밋하더라도, 현재 실행 중인 쿼리는 시작 시점의 스냅샷 데이터를 조회한다.
- 비차단 읽기(Non-blocking Read) : 데이터를 읽을 때 공유 잠금을 사용하지 않는다. 그래서 누군가 데이터를 수정 중이라도 읽기 작업은 대기 없이 즉시 수행된다.
- 이외 추가 격리 수준 : **SERIALIZABLE**, **READ ONLY** 지원

#### 장점
- 높은 동시성 : 읽기와 쓰기가 서로를 방해하지 않음
- 리소스 효율성 : 필요한 데이터만 Undo 영역에서 가져와 락 오버헤드가 적다.

#### 문제점
- Non-Repeatable Read : 하나의 트랜잭션 내에서 같은 SELECT 문을 두 번 실행했을 때, 그 사이에 다른 트랜잭션이 데이터를 커밋하면 두 결과과 서로 다를 수 있다.
- Phantom Read : 범위 조회를 할 때, 다른 트랜잭션이 데이터를 추가하고 커밋하면 이전에 없던 레코드가 생길 수 있다.]

### PostgreSQL
PostgreSQL은 표준 SQL을 가장 잘 준수하면서도, 매우 정교한 MVCC 매커니즘을 가진 데이터 베이스이다. 기본 격리 수준은 **READ COMMITTED**이다.

- 특징 : 오라클과 유사하게 커밋된 데이터만 읽는다. 하지만 내부적으로 모든 행의 버전을 관리하여 읽기/쓰기 간의 간섭이 전혀 없다.
- 장점 : 데드락 발생 확률이 낮고, 일반적인 웹 서비스 환경에서 성능과 데이터 일관성의 규형이 매우 뛰어나다.
- 단점 : Phantom Read나 Non-Repeatable Read가 발생할 수 있어, 높은 정합성이 필요한 로직은 별도의 처리가 필요하다.

### SQL Server
SQL Server(MS-SQL)는 데이터의 정확성과 참조 무결성을 매우 중요하게 여기는 엔진이다. 타 DBMS와 달리 **잠금(Lock) 기반**의 동시성 제어를 기본으로 사용하기 때문에, 격리 수준에 따른 성능 차이가 명확하게 드러난다.

기본 격리 수준은 ```READ COMMITED```이다.
- 특징 : 데이터를 읽을 때 공유 잠금을 걸고, 읽기가 끝나면 해제한다.
- 장점 : 커밋되지 않은 데이터를 읽지 않아 데이터의 신뢰도가 높다.
- 단점 : 읽기 작업이 쓰기를 차단하거나, 쓰기 작업이 읽기 작업을 차단하는 **Blocking** 현상이 잦다.

#### RCSI
SQL Server는 기본 ```READ COMMITED```의 블로킹 문제를 해결하기 위해 행 버전 관리 기술을 추가로 제공한다.
- 원리 : ```READ COMMITED``` 수준에서 작동하되, 오라클처럼 tempdb에 데이터의 이전 버전을 저장하여 읽기 시 잠금을 걸지 않게 한다.

### Next-Key Lock
Next-Key Lock은 MySQL의 InnoDB 스토리지 엔진에서 사용하는 잠금 메커니즘이다. 핵심은 레코드 잠금과 간격 잠금을 합진 형태라는 점이다.

1. Record Lock (레코드 락)
    - 인덱스 레코드 자체에 잠금을 거는 락
    - 예 : ```WHERE id = 5``` 인 행을 수정할 때 id=5인 레코드에 락을 건다.

2. Gap Lock (간격 락)
    - 레코드와 레코드 사이의 빈 공간을 거는 락
    - 예 : ```id=5```와 ```id=10``` 사이에 새로운 데이터가 삽입되지 않도록 그 사이 범위를 잠근다.

즉, ```Next-Key Lock = Record + Gap Lock```으로 특정 레코드를 잠그면서 그 바로 앞의 간격까지 함께 잠그는 방식이다.

- 장점 : 장점은 높은 신뢰도 DB 복제 구조 사용시 안정성을 준다.
- 단점 : 데드락 위험성과 동시성 저하가 발생한다.

!!! danger "Next-Key Lock 사용시 주의"
만약 ```WHERE``` 절에 사용한 컬럼에 인덱스가 없다면, InnoDB는 조건에 맞는 레코드를 찾기 위해 전체 테이블을 스캔하며 모든 간격을 Next-Key Lock을 걸어버릴 수 있다. 이는 사실상 테이블 전체가 잠기는 것과 같아 서비스 장애로 이어질 수 있으므로, 반드시 인덱스 설계를 철저히 해야 한다.

### MVCC (Multi-Version Concurrency Control)
다중 버전 동시성 제어란 데이터베이스에서 읽기 작업과 쓰기 작업이 서로를 방해하지 않도록 하기 위해 사용하는 비관적 잠금의 대안 기술이다.

쉽게 말해, 하나의 데이터에 대해 여러 개의 버전을 관리하여 동시성을 높이는 방식이다. 데이터를 수정할 때 기존 데이터를 덮어쓰는 것이 아니라, 이전 버전의 데이터를 별도의 공간(MySQL의 경우 Undo Log)에 복사해 둔다.
- Write 중일 때: 트랜잭션 A가 특정 행을 수정하고 있어도, 아직 커밋되지 않았다면 다른 트랜잭션은 락을 기다리지 않는다.
- Read 수행 시 : 트랜잭션 B는 수정 중인 데이터 대신 Undo Log에 보관된 이전 버전을 읽는다.

결과적으로 읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다는 원칙이 실현된다.

- 장점 : 높은 동시성과 일관된 읽기가 보장된다. 또한, 데드락이 감소한다.
- 단점 : 추가 저장 공간 필요, 시스템 부하(오래된 버전을 주기적으로 삭제), 트랜잭션에 따른 관리비용 증가
- Snapshot Isolation 문제 : 두 트랜잭션이 동시에 서로의 데이터를 참조하여 수정할 때 발생하는 Write Skew(쓰기 왜곡) 같은 미묘한 정합성 문제가 발생한다.

### DBMS별 격리 수준(Isolation Level) 및 특징 정리
| DBMS | 기본 격리 수준 (Default) | 특징 및 장점 | 단점 및 문제점 | 추가 지원 기술 |
| :--- | :--- | :--- | :--- | :--- |
| **MySQL (InnoDB)** | **REPEATABLE READ** | MVCC를 통해 고성능 읽기 일관성 보장. 읽기 시 잠금을 사용하지 않음. | 긴 트랜잭션 시 Undo Log 누적으로 인한 스토리지 부하 및 성능 저하 가능성. | **Next-Key Lock** (Gap Lock + Record Lock으로 Phantom Read 방지) |
| **Oracle** | **READ COMMITTED** | 가장 대중적인 수준. Undo 세그먼트를 이용해 읽기와 쓰기가 서로를 블로킹하지 않음. | 동일 트랜잭션 내 재조회 시 결과가 달라지는 **Non-Repeatable Read** 발생 가능. | **Undo Segment** 기반 MVCC, SELECT FOR UPDATE |
| **PostgreSQL** | **READ COMMITTED** | MVCC 기반으로 높은 동시성 제공. 정합성과 성능의 균형이 매우 뛰어남. | 높은 격리 수준이 필요한 경우 **Serialization Error** 처리를 위한 재시도 로직 필요. | **SSI** (Serializable Snapshot Isolation), 행 버전 관리 |
| **SQL Server** | **READ COMMITTED** | 공유 잠금(Shared Lock)을 사용하여 데이터의 참조 무결성을 강력하게 유지함. | 읽기 작업이 쓰기 작업을 차단(Blocking)하여 대규모 트래픽 시 병목 현상 발생. | **RCSI** (Read Committed Snapshot Isolation), Snapshot Isolation |
