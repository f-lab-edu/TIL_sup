# 트랜잭션 전파

트랜잭션 전파는 **이미 트랜잭션이 존재하는 상태에서 다른 트랜잭션 메서드가 호출될 때, 그 호출을 어떻게 처리할지 정하는 규칙**이다.  
즉, **기존 트랜잭션에 참여할지**, **새로운 트랜잭션을 시작할지**, 혹은 **트랜잭션 없이 실행할지**를 결정하는 옵션이다.

예를 들어 `orderService.placeOrder()` 안에서 `paymentService.pay()`를 호출한다고 하자.  
이때 `pay()`에도 `@Transactional`이 붙어 있다면, 스프링은 다음을 판단한다.

- 이미 실행 중인 트랜잭션에 참여할지
- 새로운 트랜잭션을 시작할지
- 트랜잭션 없이 실행할지

즉, 트랜잭션 전파는 **트랜잭션 경계가 중첩될 때의 참여 규칙**이라고 볼 수 있다.

---

## 트랜잭션 전파를 알아야 하는 이유
메서드마다 @Transactional을 붙였다고 해서, 항상 각각 독립적으로 동작하는 것이 아니다.   
즉, 서비스 메서드가 서로 호출될 때 기존 트랜잭션에 같이 묶일지, 새 트랜잭션을 만들지, 트랜잭션 없이 실행할지를 이해하지 못하면 의도와 다른 전혀 다른 커밋/롤백이 발생할 수 있다. 또한, 성능에 대한 이슈도 있을 수 있다. 그러므로 전파 속성을 잘 이해하고 있어야 한다.

트랜잭션 전파를 알아야 하는 이유를 더 자세하게 알아보자.

## 스프링 @Transactional 전파 옵션

### 1. REQUIRED

가장 기본이고, 실무에서도 가장 많이 사용하는 전파 옵션이다.

- 기존 트랜잭션이 있으면 그 트랜잭션에 참여
- 기존 트랜잭션이 없으면 새 트랜잭션 시작

즉, 바깥 메서드와 안쪽 메서드가 **하나의 트랜잭션**처럼 동작한다.

```java
@Transactional
public void order() {
    paymentService.pay(); // 기존 트랜잭션 참여
}
```

이 경우 order()와 pay()는 같은 트랜잭션 안에서 실행된다.
따라서 안쪽 메서드에서 예외가 발생하면 바깥 메서드까지 함께 롤백될 수 있다.

### 2. REQUIRES_NEW

항상 새로운 트랜잭션을 시작하는 옵션이다.
	•	기존 트랜잭션이 있으면 잠시 중단
	•	현재 메서드는 새 트랜잭션에서 실행
	•	실행이 끝나면 다시 바깥 트랜잭션으로 복귀

즉, 바깥 트랜잭션과 안쪽 트랜잭션이 완전히 분리된다.

```
@Transactional
public void order() {
    logService.saveLog();
}
```

```
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog() {
    ...
}
```

이 경우 saveLog()는 독립적인 트랜잭션으로 실행된다.
따라서 바깥 order()가 나중에 롤백되더라도, saveLog()가 이미 커밋되었다면 로그는 그대로 남을 수 있다.

이 방식은 다음과 같은 경우에 유용하다.
	•	감사 로그 저장
	•	이력 기록
	•	실패해도 본 로직과 분리해서 남겨야 하는 작업

다만 핵심 비즈니스 데이터에 무분별하게 사용하면 데이터 일관성이 깨질 수 있으므로 주의해야 한다.

### 3. NESTED

NESTED는 기존 트랜잭션 안에서 savepoint를 만드는 방식이다.
	•	바깥 트랜잭션은 유지
	•	안쪽 메서드는 savepoint 기준으로 부분 롤백 가능

즉, 완전히 새로운 트랜잭션을 만드는 것이 아니라 부모 트랜잭션 안의 하위 작업 단위를 만드는 개념에 가깝다.

특징은 다음과 같다.
	•	전체 트랜잭션은 유지됨
	•	내부 작업만 부분적으로 롤백 가능
	•	DB와 트랜잭션 매니저가 savepoint를 지원해야 함

실무에서는 REQUIRED, REQUIRES_NEW보다 사용 빈도는 낮다.

### 4. SUPPORTS
	•	기존 트랜잭션이 있으면 참여
	•	없으면 트랜잭션 없이 실행

즉, 있으면 같이 타고, 없으면 그냥 실행하는 방식이다.

주로 조회성 로직에서 고려할 수 있지만, 의도를 모호하게 만들 수 있어 남용하진 않는 편이다.

### 5. MANDATORY
	•	기존 트랜잭션이 반드시 있어야 함
	•	없으면 예외 발생

즉, 이 메서드는 반드시 어떤 트랜잭션 안에서만 호출되어야 한다는 강한 제약을 걸 때 사용한다.

### 6. NOT_SUPPORTED
	•	기존 트랜잭션이 있으면 중단
	•	트랜잭션 없이 실행

즉, 일부러 비트랜잭션 상태로 실행하고 싶을 때 사용한다.

예를 들면 다음과 같은 경우가 있다.
	•	오래 걸리는 조회
	•	외부 API 호출
	•	트랜잭션 범위 안에 묶고 싶지 않은 작업

### 7. NEVER
	•	트랜잭션이 있으면 예외 발생
	•	반드시 트랜잭션 없이 실행

NOT_SUPPORTED보다 더 강한 제약이다.
즉, 트랜잭션이 절대 존재하면 안 되는 메서드에 사용한다.

## 많이 사용하는 옵션

실무에서는 아래 3가지를 많이 사용하고, 가장 많이 사용하는 것은 REQUIRED이다.

	•	REQUIRED
	•	REQUIRES_NEW
	•	NESTED

대부분의 비즈니스 로직은 REQUIRED 옵션으로 충분하다.  
기존 트랜잭션이 있으면 참여하고, 없으면 새로 시작한다.  
사실 거의 REQUIRED로 동작하고, 만약 다른 옵션을 사용해야 한다면 해당 로직이 책임 분리가 덜 되었을 수도 있다.

### 주의할 점

REQUIRES_NEW는 편리해 보이지만 잘못 사용하면 위험하다.

예를 들어 주문, 결제, 재고처럼 같이 성공하거나 같이 실패해야 하는 핵심 로직에 REQUIRES_NEW를 사용하면, 일부만 먼저 커밋되어 데이터 정합성이 깨질 수 있다.

따라서 전파 옵션은 다음 기준으로 선택하는 것이 좋다.
	•	핵심 비즈니스 로직: 보통 REQUIRED
	•	독립적으로 남겨야 하는 부가 작업: REQUIRES_NEW
	•	부분 롤백 필요: NESTED 검토

또한 전파 속성은 **성능 측면에서도 영향**을 줄 수 있다.

기본값인 `REQUIRED`는 여러 작업이 하나의 트랜잭션으로 묶이기 쉽기 때문에, 트랜잭션이 길어질수록 락 점유 시간도 함께 길어진다. 그 결과 다른 요청이 동일한 데이터를 기다리게 될 수 있고, 심한 경우 데드락 가능성도 높아질 수 있다.

반대로 `REQUIRES_NEW`를 과도하게 사용하면, 새로운 트랜잭션이 반복적으로 생성되고 종료되면서 추가 오버헤드가 발생할 수 있다. 또한 트랜잭션 경계가 지나치게 잘게 나뉘면 흐름이 복잡해져 디버깅과 장애 분석도 어려워질 수 있다.

결국 전파 속성으로 인한 문제는 단순한 설정 실수라기보다, 서비스 책임과 트랜잭션 경계를 어떻게 나눴는지와 연결되는 경우가 많다. 그래서 각 속성의 의미를 정확히 이해하고, 현재 로직이 어떤 성격의 작업인지 먼저 판단하는 것이 중요하다.
